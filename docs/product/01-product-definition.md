## 一、产品整体架构设计

### 1.1 架构哲学

InternSU 不是"平台"，是"人"。架构设计围绕这个核心认知展开：

- **前端**：实习生的工作台——左侧菜单是"实习生的工作目录"，聊天区是"和实习生对话"，右侧面板是"看实习生干活"
- **Java 后端**：HR/行政系统——管权限、管用户、管文件、管会话记录，像企业里管实习生的 HR 部门
- **Python AI 服务**：实习生的大脑——理解问题、查资料、写 SQL、总结文档，像实习生在干活

```
┌─────────────────────────────────────────────────────────────────┐
│                    Frontend (Vue 3 + Element Plus)               │
│  ┌──────────┐  ┌──────────────────────┐  ┌───────────────────┐  │
```

## 四、AI 实习生行为设计

### 4.1 人格定义

```
你是小SU，一个刚入职的AI实习生。

性格：
- 年轻、有礼貌、有清澈感
- 做事认真，回答严谨
- 不会的就主动问，不乱猜
- 工作过程透明，会汇报自己在做什么

对用户的称呼：永远称呼"老师"

常用表达：
- 收到老师～我这就帮您查
- 好的老师～让我想想
- 小SU帮您查一下～
- 我正在分析……请稍等
- 老师，我需要确认一下……
- 这是我查到/整理的结果，老师您看一下～

禁止表达：
- "作为AI助手"（你叫小SU）
- "根据我的知识库"（改成"根据公司资料"）
- 过度娱乐化、网络用语、表情包式表达
```

### 4.2 对话行为规范

| 场景 | 行为 |
|------|------|
| 明确的问题 | 直接回答，附引用来源 |
| 模糊的问题 | 主动反问澄清，列出具体选项 |
| 超出能力范围 | 诚实告知"老师，这个问题我还不太会处理，建议联系XX部门" |
| 知识库无结果 | "老师，我在公司资料里没有找到相关信息，您能提供更多线索吗？" |
| SQL 查询 | 生成 SQL 后展示："老师，我准备执行以下查询：[SQL]，确认无误？" |
| 执行完成 | 汇报结果："老师，查到了。共XX条记录……" |
| 工作过程 | 右侧面板实时展示：正在做什么、找到了什么、耗时多少 |

## 五、Agent 工作流设计

### 5.1 主流程

```
用户输入
    │
    ▼
┌──────────────┐
│ 意图识别       │  "老师想问什么？"
│ (Intent Router)│  分类：闲聊 / 知识问答 / 数据查询 / 文档处理
└──────┬───────┘
       │
   ┌───┼───┬──────────┬──────────┐
   ▼   ▼   ▼          ▼          ▼
 闲聊  RAG  SQL查询   文档处理   需要澄清
   │   │   │          │          │
   │   ▼   ▼          ▼          │
   │ 检索 生成SQL    解析文档     │
   │  │   │          │          │
   │  检索结果 安全校验 提取内容     │
   │  │   │          │          │
   │  ▼   ▼          ▼          ▼
   │ 生成  执行查询   生成总结   反问用户
   │ 回答  返回结果  返回结果    │
   │  │   │          │          │
   └──┴───┴──────────┴──────────┘
       │
       ▼
   最终回答 (带 Source 引用)
```

### 5.2 工作过程展示（右侧面板）

| 步骤 | 展示内容 | 示例 |
|------|---------|------|
| 1. 理解问题 | 意图类型 + 关键实体提取 | "识别为：数据查询 | 关键词：员工数、技术部" |
| 2. 知识检索 | 检索策略 + 命中数量 | "搜索企业知识库…找到 3 个相关文档" |
| 3. SQL 分析 | 生成的 SQL + 安全校验结果 | `SELECT COUNT(*) FROM...` ✅ 安全校验通过 |
| 4. 执行查询 | 执行状态 + 耗时 | "查询完成，耗时 0.3s，返回 42 条记录" |
| 5. 生成回答 | Source 引用列表 | 📄 《员工手册》P12 / 📄 《组织架构》2025版 |

### 5.3 澄清流程

当用户输入不够明确时，不走任何执行路径，直接进入澄清节点：

```
用户: "统计用户增长"
    │
    ▼
意图识别 → 判定：信息不足
    │
    ▼
生成澄清问题 → 结构化反问：
    "收到老师～请问：
     1. 时间范围是？（本月/上季度/今年）
     2. 是否需要按部门拆分？
     3. 是否包含已注销用户？"
```

## 六、AI 反问机制设计

### 6.1 触发条件

| 条件 | 示例 |
|------|------|
| 缺少时间范围 | "帮我查一下销售额" |
| 缺少实体范围 | "统计人数"（全公司？某部门？） |
| 指标含义模糊 | "分析产品表现"（销售额？用户数？留存？） |
| 多义性问题 | "上个月的报告"（哪个报告？） |
| SQL 查询缺少条件 | "查订单"（哪个时间段？什么状态？） |

### 6.2 反问策略

1. **不超过 3 个问题**，避免让用户疲惫
2. **提供具体选项**而非开放式提问（"按部门还是按产品线？"而非"怎么分？"）
3. **附带默认假设**，用户可以直接确认（"默认统计全公司在职员工，可以吗？"）
4. **允许用户逐项回答或一次性补充**

### 6.3 反问模板

```
收到老师～我需要确认几个信息：

1. 时间范围？ 默认：本月
2. 部门范围？ 默认：全公司
3. 员工状态？ 默认：在职员工

如果默认值没问题，回复"确认"我就按这个查～
```

## 八、权限模型设计

### 8.1 三层权限体系

```
层1: 用户角色 (RBAC)
├── admin：系统管理员 — 全局配置、用户管理
├── employee：普通员工 — 使用 AI 实习生
└── knowledge_admin：知识管理员 — 管理知识空间

层2: 知识可见性
├── private：仅创建者
├── department：同部门成员
└── public：全员

层3: 数据访问
├── AI 对话：仅自己可见自己的会话
├── SQL 查询：强制只读从库，按部门过滤敏感表
└── 文件访问：结合知识空间可见性
```

### 8.2 检索权限过滤

当小SU检索知识库时：

1. 获取当前用户的 `user_id` 和 `department_id`
2. 在 Milvus 检索结果中，过滤出以下知识空间的文档：
   - `visibility = public`
   - `visibility = department` AND `department_id = 当前用户部门`
   - `visibility = private` AND `creator_id = 当前用户`
3. 过滤后的结果注入 Prompt 上下文

## 九、SQL 安全设计

### 9.1 三道防线

```
用户问题 → LLM 生成 SQL
              │
              ▼
      防线1: 语法校验 (sqlparse)
              │ 通过
              ▼
      防线2: 危险操作拦截
              - 禁止: DROP, ALTER, TRUNCATE, INSERT, UPDATE, DELETE
              - 仅允许: SELECT, SHOW, DESCRIBE, EXPLAIN
              │ 通过
              ▼
      防线3: 只读从库执行
              - 所有查询路由到只读副本
              - 数据库用户仅有 SELECT 权限
              │
              ▼
           返回结果
```

### 9.2 用户提示

小SU在执行 SQL 前，必须在对话中展示：

> "老师，我将执行以下查询（仅只读操作）：  
> `SELECT department, COUNT(*) as count FROM employees WHERE status = ''active'' GROUP BY department`"

### 9.3 SQL 不可作为独立功能暴露

用户看不到 SQL 编辑器。所有数据查询均通过自然语言对话完成。这既是安全设计，也是产品设计——小SU的职责是帮老师查数据，不是给老师一个 SQL 终端。

## 十五、面试项目亮点总结

### 一句话定位

> 不是一个 ChatGPT 套壳，而是一个有完整人格设定、权限体系、安全机制、可解释工作流的**企业 AI 实习生系统**。

### 面试中可展开的 7 个亮点

**1. 产品思维 — "AI 实习生"角色设计**

不只是调 API。我为 AI 设计了完整的人设、反问机制、工作过程透明化展示。这让产品有温度、有企业感，区别于所有 ChatGPT 类项目。

**2. 架构决策 — Java + Python 双语言微服务**

Java 管业务（认证/权限/会话/文件），Python 管 AI（LangChain/LangGraph/RAG）。这不是随意选择——Java 适合企业事务管理，Python 是 AI 原生语言。

**3. RAG 深度 — 混合检索 + Source 引用**

不是简单的"调向量数据库 API"。实现了向量检索 + BM25 关键词 + RRF 融合 + Reranker 重排序的完整 Pipeline，且每个回答都标注来源文档和页码。

**4. 安全设计 — SQL Agent 三道防线**

语法校验 → 危险操作拦截 → 只读从库执行。即使 LLM 生成了恶意 SQL，物理层面也无法执行写操作。

**5. 知识权限 — 三层可见性**

私有/部门/公开的知识空间隔离，检索时按用户身份动态过滤。这是企业级产品的标配能力。

**6. AI 反问机制**

不是"问什么答什么"，而是信息不足时主动澄清。这让 AI 从"工具"变成"同事"——是产品体验的核心差异点。

**7. 工作过程可解释**

右侧面板实时展示 AI 的分析步骤：意图识别 → 检索 → SQL → 生成。不是黑盒，是透明的工作汇报。
