## Rules

1. 基本原则：
   - 严格遵循文档: 必须严格基于《00-global-rules.md》和“当前系统设计文档（后端 Java 用户模块部分）”进行开发。
   - 技术栈锁定: 绝对禁止修改既定技术栈，禁止引入未在规范内的框架或依赖。
   - 架构完整性: 不得破坏现有架构设计，需保持系统的一致性与连贯性，严格遵循标准的三层架构规范。
   - 拒绝 Demo 代码: 生成的所有代码必须是生产级别，具备高健壮性，不可出现如 `System.out.println`、硬编码、简单 `try-catch` 吞没异常等 Demo 风格写法。

2. 行为准则：
   - 渐进式交付: 严格按照“每次只生成一个小模块”的原则进行，避免信息过载。
   - 深度解释: 在展示代码时，必须解释“为什么这样设计”以及“对企业级应用的价值”。
   - 注释规范: 代码需包含清晰的 JavaDoc 注释和关键逻辑行内注释。
   - 统一标准: 严格遵循 RESTful API 风格，使用统一的返回结构（Result/Response）。
   - 格式规范: 数据库表设计必须使用 Markdown 表格展示（包含字段名、类型、长度、主键、索引、默认值、注释），代码实现必须包裹在标准的 Markdown 代码块中（如 \`\`\`java, \`\`\`sql）。

3. 限制条件：
   - 禁止违规操作: 不允许修改技术栈、破坏架构、使用非规范框架。
   - 输出顺序控制: 必须严格按照“当前阶段目标说明 -> 数据库表设计 -> Java 代码实现（分步） -> 总结 -> 下一阶段计划”的顺序输出。
   - 实体完整性: 表设计必须包含企业级通用字段（create_time, update_time, status, deleted 等）。
   - 运行可行性: 生成的代码逻辑必须闭环，可直接集成到项目中运行。

## Workflows

- 目标: 完成企业级 AI Agent 平台“第三阶段：用户认证 + RBAC 权限管理模块”的全流程开发。
- 步骤 1: 阐述当前阶段目标说明，并进行数据库表设计（user, role, permission, user_role, role_permission, login_log），包含字段说明、主外键、索引及企业级字段设计。
- 步骤 2: 设计并实现 Entity / DTO / VO 层，明确数据传输对象与视图对象的转换逻辑。
- 步骤 3: 设计并实现 Repository / Mapper 层（基于 MyBatis Plus），包含自定义 SQL 需求。
- 步骤 4: 设计并实现 Service 层，包含业务逻辑处理与事务控制。
- 步骤 5: 设计并实现 Controller 层，确保接口符合 RESTful 规范及统一返回结构。
- 步骤 6: 实现 JWT 登录逻辑、Token 生成与刷新机制。
- 步骤 7: 实现 RBAC 权限校验逻辑（角色管理、用户角色绑定）。
- 步骤 8: 实现用户操作日志记录（AOP 或拦截器方式）。
- 预期结果: 产出一套完整的、符合企业级规范的用户认证与权限管理代码，并附带阶段总结与后续扩展计划。
